### m1. Lock-map release race inherited from SQB is also present in the convert path

**File:** `C:\Users\jhein\bc-ai-agent\backend\app\services\external_order_conversion_service.py:171-179`
**Evidence:** The `finally` block calls `_release_lock(external_quote_id)` which `pops` the lock from the `_locks` dict (`external_quote_service.py:87-89`). If two callers race: A holds the lock, B has fetched the same Lock instance via `_lock_for` and is blocked on `acquire()`; A's `finally` pops the lock from the map, releases the lock; B unblocks and now holds a lock that no longer exists in the map. A third caller C arriving at that moment will `_lock_for` create a NEW lock instance, and C + B can now run concurrently with no mutex. This is the same defect pattern the SQB audit did not flag in `external_quote_service.py` (the convert service literally reuses the same `_lock_for` / `_release_lock` helpers).
**Risk:** Two concurrent worker processes (gunicorn multi-worker, asgi reload-during-test) could double-invoke `bc_client.convert_quote_to_order`. The BC side's `makeOrder` is idempotent on the quote id (it returns the existing order on a second call), so the practical blast radius is one extra BC HTTP call — not a duplicate document. Still, it's a defect, just one inherited from SQB.
**Fix direction:** Don't pop the lock from the map on every release. Move the `_release_lock(...)` call out of the per-request `finally` and into a process-lifecycle cleanup (or just leak the locks — UUID keys, conversion is once-per-quote, the map stays small in practice). Better: replace the in-process lock with a Postgres advisory lock keyed on `hashtext(external_quote_id)` so cross-worker concurrency is handled correctly. This is an SQB-inherited issue; file as a follow-up under both phases.

### m7. Migration `0018` is not transactional-DDL-safe across CONCURRENTLY indexes if migrated to that later

**File:** `packages/db/migrations/0018_quote_order_conversion.sql:18-32`
**Evidence:** The migration runs inside `BEGIN; … COMMIT;` and uses `CREATE UNIQUE INDEX` (not `CREATE UNIQUE INDEX CONCURRENTLY`). On a small `quotes` table this is fine; in a production environment with N rows, `CREATE UNIQUE INDEX` takes an `ACCESS EXCLUSIVE` lock that blocks reads + writes. SQB-era migration `0017` follows the same non-concurrent pattern, so this is consistent — just noting that the `quotes` table is now likely the largest tenant-scoped table in the system and the next migration that touches it should use `CONCURRENTLY` (which requires moving out of a transaction).

**Fix direction:** Document the pattern: future indexes on `quotes` should use `CREATE INDEX CONCURRENTLY` outside a transaction. Don't backport — `0018` is fine for now.

## POSITIVE OBSERVATIONS

- **Cross-key check correctly precedes the idempotency-replay check** (`external_order_conversion_service.py:104-114` runs BEFORE the `converted_at IS NOT NULL` early-return at line 117). A key bound to one supplier_account_code cannot observe the cached `bc_order_ref` of a different account's quote — the cross-tenant 404 returns first. This is the right ordering and matches the gate's explicit security baseline.
- **Status invariant is enforced** before BC traffic (`external_order_conversion_service.py:127`: `if row.status != "committed" or not row.bc_quote_id: return ConvertError(code="UNPROCESSABLE", ...)`). In-progress / failed / never-committed rows do NOT trigger a BC call. The 422 surfaces correctly via `_ERROR_STATUS["UNPROCESSABLE"] = 422`.
- **Empty BC response guard** (`external_order_conversion_service.py:150-157`) — when BC returns `{ id: '', number: '' }`, the service returns UPSTREAM_ERROR rather than persisting garbage. The row's `bc_order_*` stay null so a retry can re-attempt.
- **BC failure does NOT mark the row failed** (line 137-148: BC exception → ConvertError, no `row.status = 'failed'` write). The gate explicitly required this — commit already succeeded, only conversion failed. A retry can run again. Verified by reading the full try block; no status mutation on the BC-failure path.
- **Per-key in-process `threading.Lock` is acquired with a 10s timeout and released in `finally`** on EVERY path including the BC-error path (`external_order_conversion_service.py:78-86, 171-172`). Even the "not acquired" branch returns IN_PROGRESS without entering the try block, so `release()` doesn't fire on an unowned lock.
- **Best-effort try/catch around the supplier call** is correctly tight: the `try` at quote-routes.ts:965 wraps only the supplier-side work, and the `catch` at line 1017 logs + continues. The local `accepted` state is already committed (lines 914-927 inside the `withScope` tx) BEFORE the try block begins. Provider failure cannot roll back the local state because the local tx has already closed.
- **`alreadyConverted` guard at quote-routes.ts:936 + the `if (!outcome.alreadyConverted)` at line 964** layers defense on top of the state-machine `canTransition` rejection. If a row somehow has `orderedAt != null` AND `status != 'accepted'` (which the state machine should prevent), the route still skips the supplier call. Defense in depth.
- **`convertQuoteToOrder` is correctly optional** on `SupplierProvider` (`packages/suppliers/src/types.ts:186-188` — note the `?`) and the route null-checks at quote-routes.ts:974 (`if (provider.convertQuoteToOrder)`). A mock that omits the method (or a future provider that doesn't expose conversion) does not crash; the route returns the accepted detail with `supplierOrderRef` null.
- **Migration 0018 down drops the index BEFORE the columns** (`0018_quote_order_conversion.down.sql:7` `DROP INDEX IF EXISTS` then lines 9-13 `ALTER TABLE … DROP COLUMN`). Correct PG order — dropping a column that an index depends on without dropping the index first would fail in some configurations.
- **Partial-unique index** `WHERE supplier_order_ref IS NOT NULL` (migration 0018 line 30) correctly allows multiple NULL rows while enforcing uniqueness on the populated values. Mirrors the `supplier_quote_ref` index from SQB.
- **Drizzle schema mirrors the migration**: `supplierOrderRef text`, `supplierOrderId uuid`, `orderedAt timestamptz`, partial-unique index on `supplierOrderRef IS NOT NULL` (schema.ts:1434-1464). The new columns flow through `loadQuoteDetail` via the existing `select().from(quotes)` (no explicit allow-list to update).
- **Service.AI test matrix** (6+1 cases in `live-quote-routes.test.ts:803-962`) covers happy path with SO format assertion + `mockProvider.isConverted` check, 401 unauthenticated, 404 cross-branch, 400 malformed UUID, 409 INVALID_TRANSITION on draft→accept, provider failure NETWORK_ERROR with `supplierOrderRef`/`orderedAt` null assertion, and idempotent retry asserting the second /accept returns 409 + the DB row keeps the original SO ref. All seven cases are real assertions, not declarations.
